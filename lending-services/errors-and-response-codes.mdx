Perch uses conventional HTTP response codes. `2xx` means it worked, `4xx` means the request was invalid, `5xx` means something went wrong on our side.

## Error response format

Every `4xx` and `5xx` response uses the same envelope:

```json
{
  "code": 422,
  "status": "UNPROCESSABLE",
  "message": "Validation failed for the submitted request.",
  "errors": [
    {
      "location": "body.applicants[0].creditScore",
      "message": "Must be an integer."
    },
    {
      "location": "body.propertyValue",
      "message": "This field is required."
    }
  ],
  "requestId": "32f1a9b2-7c8e-4b1a-9d3f-abc123def456",
  "timestamp": "2026-04-06T12:34:56.789Z"
}
```

| Field | Type | Always present | Description |
|-------|------|----------------|-------------|
| `code` | `int` | Yes | HTTP status code. Same as the response status. |
| `status` | `string` | Yes | Canonical status string (see table above). |
| `message` | `string` | Yes | Top-level summary. Safe to show to end users. |
| `errors` | `array` or `null` | No | Field-level details. Present on 422s from validation, sometimes on 409s. |
| `requestId` | `string` or `null` | No | Correlation ID. Pulled from `X-Trace-Id` or `X-Request-Id` headers, or auto-generated. Use this when reporting issues. |
| `timestamp` | `string` | Yes | UTC ISO 8601 timestamp. |

The `errors` array, when present, contains objects with two fields:

| Field | Type | Description |
|-------|------|-------------|
| `location` | `string` | Dot-delimited path to the problematic field. Examples: `body.email`, `body.filters.types[1]`, `query.page`. |
| `message` | `string` | What's wrong with that field. Always client-safe. |
