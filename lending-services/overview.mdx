Welcome to the Perch Lending Services API, the centralized engine powering mortgage offer generation across Perch's platform and partner integrations.

This API evaluates borrower profiles against lender policies and product catalogues, returning ranked mortgage offers. It considers lender eligibility, applicable interest rates, and each lender's underwriting criteria to determine what a borrower can qualify for.

<Note>All monetary values returned are in **Canadian Dollars (CAD)**.</Note>


---

## What the API Does

The Lending Services API answers one question: what mortgage offers can this borrower qualify for, and at what rate?

When a request is made, the API evaluates the borrower's profile against the full lender catalogue and returns qualifying offers. It accounts for:

- **Lender eligibility:** compatibility with the borrower's province, credit score, transaction type, property type, and requested product features
- **Rate calculation:** the applicable interest rate for each eligible lender
- **Debt servicing:** whether the borrower can service the debt based on GDS and TDS ratios assessed against each lender's underwriting criteria
- **Offer ranking:** results grouped by amortization, rate type, and term, sorted by effective rate (low to high)

## Qualified vs. Unqualified Endpoints

The API exposes two modes of offer retrieval, this guide explains the difference and provides recommendations on when to use either of them.

### Qualified (`/qualified/..`)

Qualified endpoints run the complete evaluation, including debt servicing assessment.

Returned offers have been assessed against the borrower's income, existing debts, and property costs. Debt Servicing Ratios (GDS & TDS) are computed and compared against each lender's underwriting criteria to determine which offers the borrower can service.

**Use qualified endpoints when:**
- You are presenting offers to a specific, identified borrower with known financial details
- You need to show only what a borrower is eligible for (e.g., in an application flow or rate comparison tool)
- Accuracy and lender compliance are required e.g., before a borrower proceeds to a mortgage application


**Example:** A borrower has submitted their income, existing liabilities, and property details. A request to `POST /api/v1/offers/qualified/lowest-rates` returns the best available rate per product category that the borrower qualifies for, reflecting what they could realistically be approved for.

### Unqualified (`/unqualified/..`)

Unqualified endpoints apply lender eligibility and product criteria based on borrower profile attributes (property type, province, down payment, transaction type), but do not restrict offers based on debt servicing ratios. The borrower's income and existing debts are not used to filter results.

**Use unqualified endpoints when:**
- You want to show indicative or best-available rates without tying results to a specific borrower's financial position
- You are building a rate explorer, pre-qualification widget, or marketing surface where a borrower has not yet provided full financial details
- You want to benchmark what the market looks like for a given property and product type

**Example:** A prospective borrower wants to see current mortgage rates without providing financial details. A request to `POST /api/v1/offers/unqualified/lowest-rate` with a property type, province, and down payment amount returns the best rate currently available across all eligible lenders. This is suitable for powering a rate display or pre-qualification widget without requiring any personal financial information.
